## 1. PROJECT OVERVIEW

The **public-dashboard** is a **React-based Store Management and Rating Dashboard** application. It's a frontend system that enables authenticated users to browse a collection of stores, view store details, search/filter stores by name and address, sort stores by various criteria, and submit or update star ratings for individual stores. 

**Core Purpose:**
- User authentication (login/register/change password)
- Browse and manage a public catalog of stores
- Submit and update store ratings
- Search, filter, and sort stores with dynamic data fetching
- Protected routes requiring JWT-based authentication

**Tech Stack:**
- **UI Framework:** React 19.2.6 with Vite as the build tool
- **State Management:** Redux Toolkit 2.12.0 for global state
- **HTTP Client:** Axios with JWT interceptors for API communication
- **Routing:** React Router DOM 7.17.0
- **Form Handling:** React Hook Form 7.79.0 with Yup/Zod validation
- **Server Communication:** TanStack React Query 5.101.0 (installed but not extensively utilized)

**Architecture Pattern:** **Feature-based modular architecture** with clear separation of concerns:
- **API Layer:** Centralized Axios instance with JWT token injection
- **State Management:** Redux Toolkit with async thunks for side effects
- **Routing:** Route-based protection via ProtectedRoute component
- **Pages:** Smart components that handle business logic
- **Features:** Dumb slices managing state without side effects; thunks handling async operations

## 3. CORE FUNCTIONALITIES

### **A. Authentication System**
- **Login:** POST `/auth/login` ‚Üí stores JWT token & user object in localStorage
- **Register:** POST `/auth/register` ‚Üí creates new user account
- **Change Password:** PUT `/auth/change-password` ‚Üí updates user password
- **Logout:** Clears token & user from Redux state and localStorage
- **Session Persistence:** Token/user hydrated on app load from localStorage
- **Route Protection:** Unauthenticated users redirected to login

### **B. Store Browsing & Discovery**
- **Fetch Stores:** GET `/stores` with optional query parameters
- **Parameters Supported:**
  - `name` ‚Äì filter by store name (partial match)
  - `address` ‚Äì filter by store address (partial match)
  - `sortBy` ‚Äì sort field (e.g., "name", "rating")
  - `order` ‚Äì sort direction ("asc" or "desc")
- **State Management:** Stores array, loading state, error state managed in Redux
- **Pagination:** Not explicitly implemented (assumes paginated API)

### **C. Store Ratings System**
- **Submit Rating:** POST `/ratings` with `{ storeId, rating }` (1-5 stars)
- **Update Rating:** PUT `/ratings/store/{storeId}` with `{ rating }`
- **Local State:** User's current rating input tracked in component state per store
- **Post-Action Refresh:** After rating submission/update, store list is refetched with current filters applied

### **D. Search & Filter**
- **Multi-field Search:** 
  - Store name search (debounced on Enter key)
  - Store address search (debounced on Enter key)
  - Combo search with both fields simultaneously
- **Sort Capabilities:**
  - Default sort: "name" ascending
  - Toggle sort direction by clicking sort field header
  - Maintains current filters while changing sort

### **E. User Interface Actions**
- **Logout Button:** Clears auth state and redirects to login
- **Change Password Button:** Navigates to dedicated password change page
- **Rating Badge Display:** Visual indicator of store's average rating with color coding
- **Star Input:** 5-star interactive input with hover preview

## 4. KEY COMPONENTS & MODULES

### **A. Redux Store Architecture**

#### **store.js** ‚Äì Redux Store Configuration
Reducer Map:

‚îú‚îÄ‚îÄ stores     ‚Üí storeSlice (store catalog state)
‚îî‚îÄ‚îÄ ratings    ‚Üí ratingSlice (rating submission state)

#### **`features/auth/`**
- **authSlice.js:**
  - Initial state: `user`, `token`, `isAuthenticated`, `loading`, `error`, `registerSuccess`, `passwordChangeSuccess`
  - Reducers: `logout`, `clearError`, `resetRegisterSuccess`, `resetPasswordChangeSuccess`
  - Extra reducers handle async thunk states (pending/fulfilled/rejected)
  - Persists token & user to localStorage

- **authThunk.js:**
  - `loginUser(loginData)` ‚Äì async login action
  - `registerUser(registerData)` ‚Äì async registration action
  - `changePassword(passwordData)` ‚Äì async password change action
  - Error handling extracts message from API response or provides fallback

#### **`features/stores/`**
- **storeSlice.js:**
  - Initial state: `stores` array, `loading`, `error`
  - No synchronous reducers (stateless mutations)
  - Extra reducers handle `getStores` async thunk states
  - Loads data array into `state.stores`

- **storeThunk.js:**
  - `getStores(params)` ‚Äì fetches stores with optional filtering/sorting
  - Extracts `response.data.data` (nested API response structure)
  - Error fallback message: "Failed to fetch stores"

#### **`features/ratings/`**
- **ratingSlice.js:**
  - Initial state: `loading`, `error`, `successMessage`
  - Reducer: `clearRatingState` ‚Äì resets error/success
  - Extra reducers handle `submitRating` & `updateRating` states
  - Tracks submission success with message text

- **ratingThunk.js:**
  - `submitRating({ storeId, rating })` ‚Äì POST new rating
  - `updateRating({ storeId, id, rating })` ‚Äì PUT existing rating (accepts either `storeId` or `id`)
  - POST endpoint: `/ratings`
  - PUT endpoint: `/ratings/store/{ratingId}`

### **B. API Layer**

#### **`api/axios.js`**
- Base URL: `http://localhost:5000`
- Default header: `Content-Type: application/json`
- **Request Interceptor:**
  - Injects JWT token from localStorage as `Authorization: Bearer {token}`
  - Applies to all outgoing requests if token exists
- No response interceptor (error handling delegated to thunks)
